Типы репликации: синхронная (мгновенная согласованность данных) и асинхронная (задержка в обновлении данных, но меньшее влияние на производительность).

Масштабирование приложений:

Поднять несколько инстансов (экземпляров) приложений. Это позволяет системе распределять нагрузку и автоматически перенаправлять трафик, если один из инстансов выходит из строя.

Преимущества: повышенная производительность и доступность, так как инстансы работают параллельно, а система продолжает функционировать даже при отказе одного из инстансов.

Балансировка нагрузки:

Добавить балансировщик нагрузки, который распределяет запросы между несколькими инстансами. Он следит за доступностью производительностью инстансов и может исключить из обработки сбойные инстансы.

Преимущества: предотвращает перегрузку отдельных инстансов и обеспечивает плавное распределение нагрузки, повышая отказоустойчивость.

Шардирование баз данных:

Разделение базы данных на несколько логических частей (шардов) по определённому критерию, например, по идентификатору пользователя. Каждый шард работает независимо.

Преимущества: снижает нагрузку на каждую базу данных и повышает производительность, а также позволяет изолировать сбои, ограничивая их только одним шардом.

2. В чем заключается принцип "stateless" (не хранить состояние) в архитектуре REST, И какие выгоды это принесет B построении масштабируемых веб-приложений?

Ответ:

Принцип "stateless" в архитектуре REST означает, что каждый запрос от клиента к серверу должен содержать всю необходимую информацию для обработки запроса, без неявного состояния на сервере. Это приводит к тому, что сервер не нужно поддерживать и управлять сессиями и состоянием клиента. Это способствует простоте, масштабируемости и устойчивости системы к сбоям.

6. В чем разница между URI (Uniform Resource Identifier) и URL (Uniform Resource Locator), и какие рекомендации следует соблюдать при проектировании имен ресурсов в RESTful API?

Ответ:

URI (Uniform Resource Identifier) это унифицированный идентификатор ресурса, который может быть URL (Uniform Resource Locator) или URN (Uniform Resource Name).

URL - это конкретный тип URI, который указывает на адрес ресурса в сети.

URN это уникальное имя ресурса, которое не обязательно связано с его местоположением.

При проектировании имен ресурсов в RESTful API следует следовать рекомендациям:

Имена ресурсов должны быть описательными и отражать суть ресурса.

Множественное число для именования коллекций ресурсов (например, "/users" для коллекции пользователей).

Дефисы для разделения слов в именах ресурсов (например, "/order-history").

5. Какие преимущества предоставляет формат данных JSON при обмене информацией между клиентом и сервером в REST-архитектуре?

Ответ:

JSON (JavaScript Object Notation) это легковесный и удобочитаемый формат данных, который часто используется при обмене информацией между клиентом и сервером B RESTful приложениях. Использование JSON предоставляет:

Читаемость: JSON представляет данные в виде объектов и массивов, что делает его понятным для чтения человеком.

Универсальность: JSON поддерживается большинством языков программирования и позволяет легко обмениваться данными между различными системами.

Легковесность: JSON имеет небольшой объем данных. Это позволяет снизить нагрузку на сеть и улучшить производительность при передаче данных.

Простота парсинга: JSON может быть легко преобразован из передаваемого по сети текста в структурированные данные и наоборот. ЭТО ПРОСТО ИМБА!!!

Поддержка веб-браузерами: JSON легко использовать B веб-приложениях, так как его формат включает в себя объекты и массивы, а браузеры предоставляют встроенные методы для работы с JSON.

C Поддержка RESTful архитектуры: JSON хорошо сочетается архитектурой REST, так как позволяет представлять ресурсы, их атрибуты и связи в формате, который легко интерпретировать и обрабатывать.
